Concretely, in my mind, the whole vLLM teacher model generation will be split into multiple tuples of (context token 0-i, teacher's log prob of top-K/top-P of top i). 

Once all tuples are obtained from the teacher model, it will calculate the KL divergence between its own logits, with the top-K logits of the teacher model, and use it as an additinal loss.

Hence in total, in additional to the losses in standard GRPO (KL loss, policy gradient loss), we will have this new KD loss. (or more precisely the reverse KD loss, as the tokens are generated by student instead of teacher)

## Implementation Notes

### Files changed

**`verl/trainer/ppo/onpolicy_kd.py`** — Complete rewrite. The old file had two bugs: every token ID was hardcoded to 0 (never mapped from the decoded string), and the URL appended `/v1/completions` to a base URL that already ended in `/v1`. The new file fixes both. It adds `build_teacher_payload` (uses `max_tokens=0, echo=True` so the endpoint returns logprobs over the input rather than generating new tokens), a `_token_str_to_id` helper that maps decoded token strings to IDs via the tokenizer (falling back to `encode()` if `convert_tokens_to_ids` returns the unknown-token ID), and a `score_sequences` function that fires one HTTP request per sequence in parallel using a thread pool, with configurable retry and back-off.

**`verl/workers/config/actor.py`** — Two fields added to `OnPolicyKDConfig`: `timeout_ms` (default 30 000 ms) and `max_retries` (default 2), which are forwarded to `score_sequences`.

**`verl/workers/actor/dp_actor.py`** — Three changes. The `select_keys` list in `update_policy` now picks up `teacher_topk_token_ids` and `teacher_topk_logprobs` from the batch when they are present, in addition to the legacy `onpolicy_kd_loss` key. In `_forward_micro_batch`, the non-rmpad non-fused path now clones the raw response logits (before temperature scaling) into a `_response_logits` variable when KD is enabled; this is returned alongside `log_probs` in the outputs dict. In the `update_policy` micro-batch loop, the old block that simply added a pre-computed KD loss tensor is replaced: when `response_logits`, `teacher_topk_token_ids`, and `teacher_topk_logprobs` are all present, the per-token forward KL is computed inline from the logits and teacher tensors; the resulting aggregated loss is scaled by `lambda_kd * warmup_factor` before being added to the policy loss. The warmup factor ramps linearly from 0 to 1 over `warmup_ratio * total_training_steps` steps, using `global_steps` and `total_training_steps` forwarded in `data.meta_info`.

**`verl/workers/actor/megatron_actor.py`** — Two changes. `make_minibatch_iterator` now picks up the same teacher tensor keys as dp_actor. The KD block inside `loss_func` is updated to apply `lambda_kd * warmup_factor` scaling (previously the KD loss was added with coefficient 1.0). `global_steps` and `total_training_steps` are forwarded into `meta_info` in `forward_step` so the warmup computation has the necessary information.

**`verl/workers/utils/losses.py`** — Pre-existing branch change (not added in this session). Contains `_compute_onpolicy_kd_loss`, a standalone helper for the new-style megatron worker path (`ppo_loss`) that computes forward KL from `model_output["response_logits"]` and teacher tensors stored in the batch.

**`verl/trainer/ppo/ray_trainer.py`** — A new `_attach_onpolicy_kd_loss(batch, global_steps)` method is called in the training loop immediately before `_update_actor`, when `onpolicy_kd.enable` is True. It decodes `input_ids` to strings, calls `score_sequences` against the configured teacher endpoint, and attaches the returned `teacher_topk_token_ids` and `teacher_topk_logprobs` tensors to `batch.batch`. It also sets `global_steps` and `total_training_steps` in `batch.meta_info`. On any exception, it logs a warning and continues without KD data (fail-open).

### Deviations from the original plan

**dp_actor rmpad path**: When `use_remove_padding=True`, `_forward_micro_batch` does not produce `response_logits` because padding removal makes it difficult to recover the `[B, T, V]` shape without re-padding the entire vocab dimension. KD is silently skipped (fail-open) in that configuration.

**Megatron actor — no logit-based KD**: The plan's revised design called for computing KD from raw logits inside the megatron `logits_processor`. This was not implemented because raw logits in the megatron pipeline are vocab-parallel across tensor-parallel ranks, making the gather operation non-trivial. Instead, the megatron path retains the pre-computed `onpolicy_kd_loss` batch key interface and only gains the `lambda_kd * warmup` scaling fix. Full logit-based KD inside the megatron forward pass is deferred.

**Trainer attaches raw teacher tensors, not a pre-computed scalar**: The trainer's `_attach_onpolicy_kd_loss` stores `teacher_topk_token_ids` and `teacher_topk_logprobs` in the batch. The actual per-token KD loss is computed on the fly inside the actor forward pass (dp_actor path). This means the megatron path — which still expects a pre-computed `onpolicy_kd_loss` tensor — will not benefit from teacher scoring unless a caller separately pre-computes and stores that tensor.

### Tests

**`tests/onpolicy_kd/test_kd_loss.py`** — Eight unit tests that exercise the loss math using synthetic tensors, with no network calls. They verify that forward KL is always non-negative, that KL equals zero when the teacher and student distributions agree exactly, that renormalizing teacher logprobs over the top-K support always produces a valid probability distribution summing to one, and that positions excluded by the response mask contribute zero to the loss. Two tests check the warmup schedule: one confirms the coefficient ramps linearly from zero to `lambda_kd` over the configured number of warmup steps and then stays flat, another confirms that setting `warmup_ratio=0` makes the coefficient equal to `lambda_kd` from the first step. One test confirms that when all teacher tensors are absent the actor logic produces a `None` loss rather than crashing. A final test checks that the token KD output tensor has shape `[B, T]`.

**`tests/onpolicy_kd/test_teacher_client.py`** — Thirteen unit tests that cover the teacher HTTP client using mocks. Payload tests confirm that `build_teacher_payload` sets `echo=True`, `max_tokens=0`, the correct `logprobs` count, and a default temperature of 1.0. Token ID mapping tests confirm that `_token_str_to_id` returns the correct ID for a known token, falls back to `encode()` for an unknown token, and returns 0 when the encode fallback is also empty. Response parsing tests confirm that `extract_topk_tensors_from_openai_response` returns tensors of the right shape, that it takes only the last `response_length` positions when the echo response has more tokens than the response region, that rows without teacher data are filled with zeros and negative-infinity logprobs, and that an empty choices list produces all-zero, all-minus-infinity tensors. Retry and batch tests confirm that `score_sequences` retries on connection failure and succeeds on the third attempt, returns a False valid-mask entry with zeroed tensors when all retries are exhausted, returns the correct `[B, T, K]` shapes for a batch, and constructs the URL as `{base_url}/completions` rather than `{base_url}/v1/completions`.

**`tests/onpolicy_kd/test_integration.py`** — Four tests that hit the real teacher endpoint and are automatically skipped when the `TEACHER_URL` environment variable is not set. The first checks that the returned tensors have the expected `[B, T, K]` shapes. The second checks that all finite logprob values are at most zero. The third builds a random student and verifies that the computed KD loss is positive. The fourth checks that `valid_mask` is all-True for sequences the endpoint handles successfully; if the endpoint fails it skips rather than fails, so CI is never blocked.
